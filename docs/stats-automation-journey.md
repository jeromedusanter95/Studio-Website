# Stats automation — what we built and how we got there

This document is a plain-English record of the work done to automate the
download and impression counts displayed on `index.html`, `apps.html`,
and `stats.html`. It covers the final design, every dead end we hit, and
what to do if things break in the future.

Written as a reference for future-me (Jérôme) who will look at this in a
year and wonder "why the hell is it wired this way?".

---

## TL;DR

- **Goal**: stop hand-editing `stats-data.json` every time download numbers
  move. Replace hardcoded numbers on the 3 pages with values that refresh
  automatically once a day.
- **Final architecture**: GitHub Action (cron 06:00 UTC) runs a tiny
  Node.js script that reads Google Play bulk reports from a GCS bucket,
  reads App Store Connect sales reports via JWT, appends daily rows to
  `stats-history.json`, regenerates `stats-data.json`, commits and pushes
  back. The static site reads `stats-data.json` on page load.
- **No server. No database. No REST API.** GitHub itself is the database
  and the scheduler. Frontend stays a pure static site.
- **Required secrets (6)**: `GOOGLE_SERVICE_ACCOUNT_JSON`,
  `PLAY_BULK_REPORTS_BUCKET`, `ASC_KEY_ID`, `ASC_ISSUER_ID`,
  `ASC_PRIVATE_KEY`, `ASC_VENDOR_NUMBER`.
- **What is manual and will stay manual**: ad impressions total
  (`MANUAL_IMPRESSIONS` in `storage.js`) and average store rating
  (`MANUAL_AVERAGE_RATING`). Neither has a working self-service API.

## Final architecture

```
 GitHub Actions (cron 06:00 UTC daily + manual dispatch)
        │
        ▼
 scripts/fetch-stats/index.js              (orchestrator)
   ├── sources/play.js      ── Google Play bulk reports bucket
   │                            (gs://pubsite_prod_.../stats/installs/
   │                             installs_{package}_{YYYYMM}_overview.csv)
   └── sources/appstore.js  ── App Store Connect Sales Reports
                                (JWT ES256 → gzipped TSV per day)
        │
        ▼
 stats-history.json   ← append-only, idempotent per (source, date)
 stats-data.json      ← regenerated from history, read by the frontend
        │
        ▼
 git commit + git push ──► GitHub Pages rebuild ──► site shows new numbers
```

**Idempotency**: `storage.needsFetch(history, source, date)` returns false
if any row for that `(source, date)` pair already exists. The fetcher
runs a rolling 3-day window (yesterday / D-2 / D-3) to absorb Apple's
~24h sales report lag, but it won't refetch anything it already has, so
re-runs and retries are free.

**Graceful failure**: each source wrapped in try/catch. If Play fails,
App Store still runs and vice versa. `process.exit(0)` unconditionally
so the commit step always runs with whatever succeeded.

**No-op commit avoidance**: `index.js` only rewrites `stats-data.json`
when there are actual new rows. Otherwise `git diff --cached --quiet`
in the workflow detects nothing changed and skips the commit.

## Files map

| File | Role |
|---|---|
| `index.html` | Home page. Small inline script fetches `stats-data.json` and populates `[data-app-dl]` spans (per-app counts) and the hero stat panel. |
| `apps.html` | Apps page. Same pattern with `[data-app-dl]` spans for per-app counts. |
| `stats.html` | Stats page. Unchanged design. Reads `stats-data.json` totals for the 4 KPI cards. |
| `stats-data.json` | **Regenerated by the bot.** Current frontend contract: `totals.downloads`, `totals.impressions`, `totals.published_apps`, `totals.average_rating`, `apps.<slug>.downloads`. Do not edit by hand. |
| `stats-history.json` | **The "database".** One row per (source, date, app). Append-only. Baseline rows carry `source: "baseline"` so they're never refetched. |
| `scripts/fetch-stats/index.js` | Orchestrator. Computes date window, runs both sources in parallel via `Promise.all`, merges new rows, writes files. |
| `scripts/fetch-stats/storage.js` | History I/O + aggregation. Holds `APP_SLUGS`, `MANUAL_AVERAGE_RATING`, `MANUAL_IMPRESSIONS` constants. |
| `scripts/fetch-stats/sources/play.js` | Downloads monthly CSVs from GCS, decodes UTF-16LE, matches columns by header name, extracts rows for the target dates. |
| `scripts/fetch-stats/sources/appstore.js` | Builds ES256 JWT, fetches `/v1/salesReports` per date, gunzips the response, parses the TSV, sums units per Apple ID. |
| `.github/workflows/daily-stats.yml` | The cron. Installs deps with `npm ci`, runs the fetcher, commits and pushes with a bot identity. |
| `_config.yml` | Jekyll exclude so GitHub Pages doesn't serve `scripts/`, `.github/`, `README.md`, `node_modules/`. |
| `README.md` | Setup instructions for the credentials — the user-facing doc for future installs. |

## The journey (in order of pain)

### Phase 1 — Initial plan (commit `91f99d4`)

First design had 3 sources: Google Play Developer Reporting API, App
Store Connect, and AdMob Reporting API. Plan was generous: new KPI
cards, per-platform breakdown, 30-day SVG chart, the works.

Jérôme pushed back mid-implementation with **"don't change the actual
design of my website, just replace the actual stats that are hardcoded
with dynamic value uploaded daily"**. Good call. Reverted all design
changes. The stats page stayed visually identical to before — only the
numbers inside the existing KPI cards became dynamic.

### Phase 2 — Discovery: AdMob API doesn't exist for new accounts

Tried to set up AdMob API access as planned. Turned out:

- AdMob's **Settings → API Access** page no longer exists in the UI.
- The **Linked services** tab only shows Firebase and Google Ads.
- There is no self-service way to enable AdMob API access on a modern
  account — Google gradually removed it in 2024-2025 and put it behind
  a manual application form.

**Resolution** (commit `9700067`): deleted `sources/admob.js` entirely,
added `MANUAL_IMPRESSIONS = 97473` constant in `storage.js`, removed
the `ADMOB_PUBLISHER_ID` secret, dropped from README. Ad impressions
are now a hand-edited number that lives at the top of `storage.js`.
When you want to bump it, edit the constant and push.

Setup dropped from **6 secrets → 5 secrets** and **3 APIs → 2 APIs**.

### Phase 3 — Discovery: Play Developer Reporting API doesn't expose installs

After adding the 5 secrets, the workflow ran and App Store Connect
returned 4 iOS rows immediately — that side worked on the first try.
But Google Play calls all returned HTML 404 pages (Google Frontend
"robot" 404, not a JSON error).

First suspicion: service account permission missing. Jérôme granted
**"View app information and download bulk reports (read only)"** on
the **Account permissions** tab of Play Console. Still 404.

Real cause (found via WebFetch of the Play Developer Reporting API
reference docs): **the `installsMetricSet` resource does not exist**.
The Reporting API only exposes Android Vitals metrics (crash rate, ANR
rate, slow rendering, etc.), not install counts. My initial code was
calling a URL path that doesn't exist in the API at all, which is why
Google Frontend rejected it before it reached any service.

Install counts are only available via **bulk reports** — monthly CSV
files Google Play writes to a private Cloud Storage bucket named
`pubsite_prod_<developer-account-id>`. This is the supported path for
programmatic access to install numbers.

**Resolution** (commit `68e9dd3`): rewrote `sources/play.js` from
scratch to use `@google-cloud/storage` instead of the non-existent
REST endpoint. Removed the `googleapis` dependency entirely. Added
`PLAY_BULK_REPORTS_BUCKET` as a required secret. Updated the README.

### Phase 4 — Bulk reports gotchas

Three non-obvious things about Play bulk reports:

1. **UTF-16LE encoding.** Not UTF-8. The CSV parser has to call
   `buffer.toString('utf16le').replace(/^\ufeff/, '')` or every ASCII
   char comes out as garbage interleaved with null bytes. Multiple
   sources warned about this; easy to miss.
2. **Filename format**: `stats/installs/installs_<package>_<YYYYMM>_overview.csv`.
   One file per app per month, accumulating daily rows as the month
   progresses. The `overview` suffix means "no dimension slice" which
   is what we want.
3. **Column name to use**: "Daily Device Installs" (fallback: "Daily
   User Installs" or "Install events"). Matched case-insensitively with
   substring fallback so small Google renames don't break the parser.

### Phase 5 — Permission propagation

After pushing the bulk reports rewrite, the fetcher got past the
routing and hit the bucket. Result:

```
studio-stats-bot@studio-stats.iam.gserviceaccount.com does not have
storage.objects.get access to the Google Cloud Storage object.
```

This is the **correct** error at this stage — we're reaching the
bucket and getting a proper IAM denial instead of an HTML 404. Two
possible causes:

1. **Wrong GCS permission path.** The bucket is Play-owned and uses
   old-style object-level ACLs. It shows "Unable to load roles" in the
   GCP Cloud Storage UI because we don't own the parent project. The
   "Grant access" button is not there and **this is expected** — you
   cannot grant access via GCS IAM. The only grant path is Play
   Console's "View app information and download bulk reports" checkbox,
   which Jérôme already set correctly.
2. **First-time propagation delay.** Multiple community sources report
   that Play Console grants can take from minutes to 24 hours to
   propagate into the bucket ACL. This is apparently normal Google
   behavior and entirely outside our control.

Verified Jérôme's Play Console state: Account permissions tab, bulk
reports box ticked, Save changes button greyed out (no unsaved
changes). Configuration is correct. Waiting on Google.

### Phase 6 — Parallel Hearts package name bug (commit `45208dd`)

While waiting for propagation, spotted in a Play Console screenshot
that Parallel Hearts' package name is
`com.jeromedusanter.parallelhearts` (no underscore), but I had
hardcoded `com.jeromedusanter.parallel_hearts`. The other 3 packages
match reality. Without this fix, even once propagation kicks in, 3 of
the 4 apps would work and Parallel Hearts would silently return no
rows. Pushed the fix so it's in place when bucket access starts
working.

## Troubleshooting playbook

### "Android numbers still not updating after 24h"

Likely Play Console permission grant didn't propagate. Nuclear option:

1. Play Console → Users and permissions → click
   `studio-stats-bot@studio-stats.iam.gserviceaccount.com`.
2. Top right → **Remove user**.
3. Invite new users → paste the same email.
4. Account permissions tab → tick "View app information and download
   bulk reports (read only)".
5. App permissions tab → add all 4 apps.
6. Save changes.
7. Wait 15 minutes, re-run the workflow from the Actions tab.

If that still doesn't work, fall back to a manual Android constant the
same way we did for AdMob: add `MANUAL_ANDROID_DOWNLOADS_BY_APP` to
`storage.js` and bump by hand.

### "App Store started failing"

Most common causes:

- **401 JWT error**: the `.p8` private key stored in `ASC_PRIVATE_KEY`
  has been mangled (usually by line ending conversion). Re-paste it
  from the original `.p8` file, including the `BEGIN/END` lines.
- **404 on a recent date**: Apple hasn't posted yesterday's report yet.
  Normal, the rolling 3-day window will catch it tomorrow.

### "Workflow runs but stats-data.json isn't updating"

Check the Fetch stats step logs. If you see `[main] no new rows
fetched, skipping write`, that means both sources are idempotent-no-op
for the current window (everything already in history). This is
correct behavior — nothing to commit. The file only rewrites when
there's actual new data.

### "I want to add a 5th app"

Touch all of:

1. `apps.html`: add the big-card section with a `data-app-dl="<slug>"`
   meta cell.
2. `index.html`: add an apps-teaser card with `data-app-dl="<slug>"`
   if you want it on the home page.
3. `scripts/fetch-stats/storage.js`: add the slug to `APP_SLUGS`.
4. `scripts/fetch-stats/sources/play.js`: add the `package_name: slug`
   entry to `DEFAULT_PACKAGE_MAP`.
5. `scripts/fetch-stats/sources/appstore.js`: add the `apple_id: slug`
   entry to `DEFAULT_APP_ID_MAP`.
6. In Play Console, make sure the existing `studio-stats-bot` user has
   the new app in its App permissions list (or that Account permissions
   grants global access — it does).
7. Commit, push, run the workflow.

### "I want to start over because the bot's history is messy"

Nuclear reset:

1. Delete `stats-history.json` entirely.
2. Recreate it with just the baseline rows you want (see current file
   for the format). Only `source: "baseline"` rows survive the
   idempotency checks indefinitely.
3. Commit, push, manually run the workflow. It'll refetch the last 3
   days and regenerate `stats-data.json` cleanly from the new history.

## Lessons learned (for the next integration)

1. **Verify the API exists before writing code to call it.** The whole
   Phase 3 mess would have been avoided by fetching the API reference
   page first and confirming `installsMetricSet` was a real resource.
   Writing plausible-looking URLs from API naming conventions is a
   trap, especially for Google APIs which deliberately fragment their
   surface across multiple products.
2. **Google Play stats = bulk reports bucket.** Not Reporting API, not
   Publishing API. Anyone telling you otherwise is wrong or writing
   about Android Vitals. The bucket path is the only supported one for
   install counts.
3. **UTF-16LE on Play CSVs.** If numbers look like garbage or column
   headers don't match, check the encoding before anything else.
4. **AdMob API is gated.** Don't promise AdMob automation to anyone
   without first verifying the account has API access enabled. For new
   accounts, assume it's manual.
5. **First-time Play Console grants propagate slowly.** If you set up
   a service account for bulk reports, budget a full day before you
   consider it broken.
6. **Package names and bundle IDs: extract from the source of truth.**
   I extracted them from `apps.html` store links early on, which was
   right for 3 out of 4 — but I should have verified all 4 against
   Play Console directly before hardcoding. The `parallel_hearts` vs
   `parallelhearts` typo cost an extra debugging round.
